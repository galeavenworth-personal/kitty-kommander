# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What This Is

A configuration + skills repo ŌĆö not an application. It wires together Kitty terminal, tmux, beads issue tracker, and graphviz into a four-tab cockpit for driving Claude Code agent teams.

## Install

```bash
./install.sh
```

Checks dependencies, symlinks configs to `~/.config/kitty/` and `~/.tmux.conf`, installs skills to `~/.claude/skills/`, symlinks the inspector kitten to `~/.config/kitty/inspector`, symlinks `scripts/launch-cockpit.sh` to `~/.local/bin/kitty-kommander`, writes a wrapper script to `~/.local/bin/kommander-ui` (baked with the repo path ŌĆö re-run `install.sh` after moving the checkout), verifies the cell-leader agent definition at `.claude/agents/`, and installs beads hooks via `bd setup claude --project`. Existing non-symlink files are preserved (warned, not overwritten); the `kommander-ui` wrapper carries a sentinel line so re-runs can regenerate their own output while still preserving operator-authored scripts at that path.

## Architecture

**Two layers of terminal control:**

1. **Kitty remote control** (`kitty @ --to "$KITTY_LISTEN_ON" <cmd>`) ŌĆö manages tabs, windows, sends text. Socket path varies per instance; always use `$KITTY_LISTEN_ON`, never hardcode.
2. **tmux** ŌĆö manages panes inside the Cockpit tab. Session name is `$KITTY_KOMMANDER_SESSION` (e.g. `cockpit-my-app`), derived from the project directory basename. Always use the env var, never hardcode `cockpit`.

**Four-tab layout** (defined in `config/kitty/sessions/kommander.kitty-session`):

| Tab | Name | Content |
|-----|------|---------|
| 0 | Cockpit | tmux session `$KITTY_KOMMANDER_SESSION` ŌĆö agent panes spawn here |
| 1 | Driver | Claude Code cell-leader session (launched via `scripts/launch-claude.sh`) |
| 2 | Notebooks | euporie (terminal Jupyter editor) |
| 3 | Dashboard | Split pane: `cockpit_dash.py --dag` (left) + `cockpit_dash.py --sidebar` (right) |

**Skills** (`skills/*/SKILL.md`) are Claude Code extensions symlinked into `~/.claude/skills/`. Each skill is a SKILL.md with frontmatter (name, description, trigger phrases) and reference docs. Four skills: cockpit (pane/tab management), plot (data viz), view (image/PDF preview via timg), notebook (Jupyter generation).

**Cell-leader** (`.claude/agents/cell-leader.md`) is an agent definition that coordinates up to four specialist teammates using an Intent Cell Architecture ŌĆö creates a team via `TeamCreate`, spawns teammates via the `Agent` tool with `team_name`, tracks work via beads, integrates outputs. Use `@cell-leader` to invoke explicitly, or `--agent cell-leader` for dedicated cell-leader sessions.

**Dashboard** (`scripts/cockpit_dash.py`) is a Python 3 stdlib-only script with two modes:
- `--dag`: Reads beads issues via `bd --format=json`, generates Graphviz DOT with yarn ball sprites as image nodes, pipes through `dot -Tpng` then `timg` to render inline. Falls back to styled circles when sprites are missing.
- `--sidebar`: Shows completion bar, ready queue (priority-sorted), recent git commits. Both auto-refresh every 30s.

**Inspector kitten** (`kittens/inspector/`) is a custom kitty kitten for structured terminal inspection. Provides JSON output for tab/window layout (`ls`), screen content (`text`), tmux state (`tmux`, `tmux-text`), screenshot capture (`screenshot`), desktop integration validation (`desktop`), readiness polling (`wait`), and parallel socket enumeration (`sockets`). Two invocation modes: `kitty +kitten inspector <cmd>` (inside kitty) or `python -m kittens.inspector --socket <path> <cmd>` (standalone CLI). See `kittens/inspector/DESIGN.md` for full spec.

**Sprite system** (`sprites/`) provides the visual status language for the DAG. Yarn ball node sprites are in `sprites/nodes/`, generated by `sprites/generate.py` (requires Pillow). Kitty role/state sprites go in `sprites/kitties/` (planned). See `sprites/MANIFEST.md` for the full sprite doctrine.

## Parallel Instances

Multiple kitty-kommander instances can run simultaneously on different project directories. Each instance gets its own tmux session and kitty control socket, derived from the project directory basename.

**Entry point**: `kitty-kommander /path/to/project` (symlinked to `scripts/launch-cockpit.sh`). All launch paths (desktop right-click, Nautilus context menu, F7+A) go through this script.

**Per-instance env vars** (set by `launch-cockpit.sh`, available to all child processes):

| Variable | Example | Purpose |
|----------|---------|---------|
| `KITTY_KOMMANDER_DIR` | `/home/user/my-app` | Project directory (absolute) |
| `KITTY_KOMMANDER_SESSION` | `cockpit-my-app` | tmux session name |
| `KITTY_LISTEN_ON` | `unix:/tmp/kitty-kommander-my-app` | Kitty control socket (set by kitty) |

**Slug derivation**: `basename "$dir" | tr upper lower | tr -cs alnum-hyphen hyphen`. Two projects with the same basename in different parent dirs would collide ŌĆö rename one if needed.

## Key Conventions

- **Tokyo Night palette everywhere.** Hex values in `cockpit_dash.py` `PAL` dict, tmux.conf status styles, and graphviz node colors must stay consistent. Background: `#1a1b26`, accent: `#7aa2f7`, status colors: red `#f7768e`, green `#9ece6a`, yellow `#e0af68`, grey `#565f89`.
- **timg must be invoked as `/usr/bin/timg`** ŌĆö bare `timg` may resolve to a Python wrapper that shadows the system binary.
- **beads (`bd`) is the issue tracker.** Agents create, claim, and close work items. The dashboard reads state via `bd --format=json`. The `.beads/` directory is gitignored (local state).
- **Symlink-based installation.** Changes to files in this repo immediately take effect in `~/.claude/` and `~/.config/` after install. No copy/rebuild step needed.
- **Use the inspector kitten for integration testing.** `kitty +kitten inspector ls` for tab structure, `inspector desktop` for install validation. Tests in `test/` use pytest with skip markers for headless CI.
- **Reach for LSP, not grep, on Go and CUE code.** Two paths, same idea:
  - **Go** ŌĆö use the native `LSP` tool (wired via `gopls-lsp@claude-plugins-official` at user scope; `~/go/bin/gopls` v0.21+). Operations: `findReferences`, `goToDefinition`, `incomingCalls`, `hover`. Crosses packages cleanly (`cmd/` Ōåö `internal/`) and follows into generated `*_gen_test.go` ŌĆö critical in this scenariogen codebase.
  - **CUE** ŌĆö Claude Code's plugin marketplace has no `cue-lsp` plugin (checked Apr 2026; list: clangd/csharp/gopls/jdtls/kotlin/lua/php/pyright/ruby/rust-analyzer/swift/typescript). So we talk to `cue lsp` directly via `scripts/cue-lsp.py` ŌĆö a stdlib-only JSON-RPC client. Cold start ~60ms. Subcommands: `hover | def | refs | symbols`. Crucially, `def` on a field surfaces **every sibling file that unifies into it** ŌĆö e.g. `def` on `scenarios` in `schema/cli/launch.cue` returns launch.cue + doctor.cue + reload.cue + types.cue, because CUE field unification is invisible to grep.
  - Grep is still correct for string literals, filenames, and comments. LSP is correct for symbols.

## Testing

Tests live in `test/` and use pytest. Two categories:

**Headless tests** (run anywhere, no kitty needed):
```bash
python -m pytest test/test_desktop.py test/test_dag_dot.py -v --noconftest
```
- `test_desktop.py` ŌĆö validates install surface (desktop entry, MIME, nautilus, CLI)
- `test_dag_dot.py` ŌĆö unit tests for `build_dag_dot()` pure function in cockpit_dash.py

**Integration tests** (require running kitty + display server):
```bash
python -m pytest test/ -v
```
- `test_structure.py` ŌĆö tab layout, tmux session, env var expansion (8 tests)
- `test_parallel.py` ŌĆö parallel instance socket isolation (5 tests)
- `test_visual.py` ŌĆö screenshot capture for agent visual review (4 tests)

Integration tests auto-skip when kitty or a display server is absent (via `requires_kitty` / `requires_display` markers in `test/conftest.py`). The `--noconftest` flag is needed for headless tests because `conftest.py` imports `kittens.inspector.__init__` which references kitty-internal `kittens.tui.handler`.

Screenshot artifacts are saved to `test-artifacts/` (gitignored).

## Agent Team Coordination

This project uses Intent Cell Architecture (ICA) for agent team coordination,
built on two complementary systems:

- **TeamCreate** ŌĆö team lifecycle: creation, teammate discovery, message delivery, idle management
- **beads (`bd`)** ŌĆö work tracking: items, assignment, dependencies, file reservations, audit trail

Do not use TaskCreate/TaskList/TaskUpdate for work items ŌĆö beads replaces them.

### Team Lifecycle

Create a team with `TeamCreate`, then spawn named teammates with the `Agent` tool
using the `team_name` parameter. Each teammate joins the team and gets automatic
message delivery.

```
TeamCreate(team_name="mission-name", description="what we're building")

Agent(name="builder", team_name="mission-name", prompt="[bounded order]")
Agent(name="scout", team_name="mission-name", prompt="[bounded order]")
```

- Teammates discover each other via `~/.claude/teams/{team-name}/config.json`
- Use `SendMessage(to="<name>")` for inter-agent communication ŌĆö messages deliver automatically
- Teammates go idle between turns ŌĆö this is normal. Send a message to wake them
- Launch independent teammates in parallel (multiple `Agent` calls in one message)
- Each order must include: Mission, Purpose, Constraints, expected Output, and Escalate conditions
- Team never exceeds four specialists under one leader
- Shut down with `SendMessage(to="<name>", message={type: "shutdown_request"})`

### Three-Tier Work Model (Proto / Mol / Wisp)

All work follows beads' chemistry model:

| Tier | Phase | What | Lifecycle |
|------|-------|------|-----------|
| **Proto** | Solid | Reusable workflow template (formula) | Persists forever. `bd formula list` to browse. |
| **Mol** | Liquid | Persistent tracked work (epic + subtasks) | `bd mol pour <proto>` to create. Full audit trail. |
| **Wisp** | Vapor | Ephemeral agent task execution | `bd mol wisp <proto>` to create. Squash or burn when done. |

**Flow:** Check `bd formula list` for existing patterns before planning from scratch.
Pour a Mol for the mission. Pin subtasks to agents. Individual agent work uses
Wisps (ephemeral, gitignored). Squash Wisps into digests when done. Close the Mol.

```bash
bd mol pour <proto> --var name=<mission> --json   # Proto -> Mol (persistent)
bd mol wisp <proto> --var name=<task> --json       # Proto -> Wisp (ephemeral)
bd pin <id> --for <agent-name> --start             # Assign to teammate
bd reserve <file> --for <agent-name>               # Prevent concurrent edits
bd mol squash <id> --summary "result"              # Wisp -> digest
bd mol burn <id>                                   # Discard abandoned wisp
bd mol distill <epic-id>                           # Extract proto from completed work
```

### Beads Coordination

```bash
bd ready --json                        # Unblocked work (respects dependency chains)
bd update <id> --claim                 # Atomically claim an issue
bd close <id> --reason "text"          # Complete with summary
bd dep add <child> <parent>            # Declare dependency (blocks ready queue)
bd dep tree <id> --json                # Visualize dependency tree
bd reserve <file> --for <agent>        # File reservation (prevent conflicts)
bd lock <id> --for <agent>             # Exclusive issue access
bd --actor <name> <cmd>                # Audit trail with agent identity
bd dolt push                           # Sync state (automatic via PreCompact hook)
```

### Operating Sequence

1. **Interpret intent** ŌĆö state mission, success criteria, constraints, assumptions
2. **Check existing work** ŌĆö `bd ready` for available tasks, `bd formula list` for patterns
3. **Create team** ŌĆö `TeamCreate` if complex, work alone if simple
4. **Issue bounded orders** ŌĆö spawn teammates with `Agent` + `team_name`, each gets Mission, Purpose, Constraints, Output, Escalate
5. **Track with beads** ŌĆö `bd pin` assigns, `bd reserve` prevents file conflicts, `bd dep add` declares blockers
6. **Supervise** ŌĆö check for drift, blockers, overlap via `SendMessage` (messages deliver automatically)
7. **Integrate** ŌĆö combine outputs, resolve contradictions, close the Mol
8. **Distill** ŌĆö if the work pattern is reusable, `bd mol distill` to create a Proto for next time

### Session Protocol

- `bd prime` runs automatically at session start (SessionStart hook) ŌĆö provides live workflow context
- `bd dolt push` runs automatically before context compression (PreCompact hook)
- Use `bd` for work tracking (not TodoWrite or markdown TODOs). Use TeamCreate/SendMessage for team lifecycle
- Teammates check `bd ready` for available work, not TaskList
- Work is not complete until `git push` succeeds:
  ```bash
  git pull --rebase && bd dolt push && git push
  ```

## Beads Issue Tracker

This project uses **bd (beads)** for issue tracking. Run `bd prime` to see full workflow context and commands.

### Quick Reference

```bash
bd ready              # Find available work
bd show <id>          # View issue details
bd update <id> --claim  # Claim work
bd close <id>         # Complete work
```

### Rules

- Use `bd` for ALL task tracking ŌĆö do NOT use TodoWrite, TaskCreate, or markdown TODO lists
- Run `bd prime` for detailed command reference and session close protocol
- Use `bd remember` for persistent knowledge ŌĆö do NOT use MEMORY.md files

## Session Completion

**When ending a work session**, you MUST complete ALL steps below. Work is NOT complete until `git push` succeeds.

**MANDATORY WORKFLOW:**

1. **File issues for remaining work** - Create issues for anything that needs follow-up
2. **Run quality gates** (if code changed) - Tests, linters, builds
3. **Update issue status** - Close finished work, update in-progress items
4. **PUSH TO REMOTE** - This is MANDATORY:
   ```bash
   git pull --rebase
   bd dolt push
   git push
   git status  # MUST show "up to date with origin"
   ```
5. **Clean up** - Clear stashes, prune remote branches
6. **Verify** - All changes committed AND pushed
7. **Hand off** - Provide context for next session

**CRITICAL RULES:**
- Work is NOT complete until `git push` succeeds
- NEVER stop before pushing - that leaves work stranded locally
- NEVER say "ready to push when you are" - YOU must push
- If push fails, resolve and retry until it succeeds
